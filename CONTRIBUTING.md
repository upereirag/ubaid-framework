# Contributing to UBAID Framework

The UBAID Framework is published in draft for community adversarial review. Contributions are welcomed and structured into two channels with different purposes.

## How to Contribute

### Channel 1 — GitHub Issues

Use Issues for:

- Pointing out inconsistencies, contradictions, or ambiguities in the framework
- Suggesting specific corrections or clarifications to existing definitions
- Reporting cases where the framework appears to fail or produce unintuitive results
- Proposing minor extensions tied directly to existing framework elements
- Asking conceptual questions that may improve the documentation

Issues should be specific, reference the exact section of the whitepaper they address, and propose a concrete change or articulate a concrete concern.

### Channel 2 — Structured Contribution Form

Use the form for substantive contributions:

- Critical reviews of the framework as a whole or major sections
- Proposed new satellite vectors for existing celestial bodies
- Documented Black Hole observations
- Proposed new use cases or case studies applying the framework
- Mathematical formalization proposals
- Adjacent research connecting UBAID to existing literature

The form is available at: [forms.gle/cgTKUg3bLz7pPvgw8](https://forms.gle/cgTKUg3bLz7pPvgw8)

All contributions submitted through the form are subject to the [Contributor License Agreement](CLA.md).

## What Makes a Strong Contribution

The framework benefits most from contributions that:

- Reference specific sections and apply precise critique rather than general impressions
- Propose alternative formulations rather than merely identifying problems
- Test the framework against concrete cases — real incidents, hypothetical scenarios, edge cases
- Bring expertise from adjacent fields: philosophy of mind, law, security research, AI alignment, computational physics
- Engage with the framework's own logic rather than rejecting its premises (which is a separate, legitimate but distinct contribution)

```

```

**Conceptual Concern**

```

Concern:
Adjacent literature this relates to:

```

**Use Case Proposal**

```
Real or hypothetical incident:
Framework elements applied:
Results obtained:
Where the framework worked / where it struggled:
```

## Code of Conduct

Engagement with the framework should be:

- **Rigorous** — substantive critique is welcomed and prioritized over polite agreement
- **Specific** — vague impressions are less useful than concrete claims
- **Respectful** — the framework is published in draft and the author treats critique as collaboration
- **Attributed** — credit for accepted contributions is given in the changelog and acknowledgments

The framework's own thesis is that no single agent can attribute responsibility correctly without distributed knowledge. The same principle applies to its own development. Adversarial review is not optional — it is structural.

---

## What Happens to Your Contribution

1. The author reviews each submission within a reasonable timeframe
2. Accepted contributions are incorporated into the next version with attribution
3. Contributions that propose major restructuring are discussed in GitHub Discussions before incorporation
4. Rejected contributions receive a written rationale
5. All accepted contributors are listed in the framework's acknowledgments
